# 13. 把项目知识接入 Agent

## 这一章解决什么问题

到目前为止，agent 已经具备了不少通用能力：

- 能理解任务
- 能在仓库里搜索和读取
- 能做 planning
- 能在必要时拆 subagent

但真实项目里最常见的问题并不是：

- agent 完全不会写代码

而是：

- agent 懂通用编程，却不懂当前仓库

这一章要解决的，就是如何把项目知识接入 agent。

## 为什么项目知识很重要

很多真实任务失败，并不是因为模型不会写代码，而是因为它不了解：

- 目录约定
- 架构原则
- 代码风格
- 测试规范
- 团队工作方式

这些东西对人类团队来说往往是“默认常识”。

但对 agent 来说，如果你不显式提供，它就只能猜。

而仓库内最危险的错误，往往不是“完全错误”，而是：

- 看起来合理
- 放进这个项目却不对

所以项目知识的价值，不在于“多给一点背景”，而在于：

**让 agent 从通用正确，走向仓库内正确。**

## 先记住一句话

**项目知识不是附加说明，而是让 agent 真正成为团队合作者的前提。**

如果一个 agent 只知道：

- TypeScript 最佳实践
- React 通用写法
- Node.js 通用模式

它最多只能算一个通用编程助手。

只有当它开始理解：

- 这个仓库怎么组织
- 这个团队怎么写代码
- 这个系统有哪些约束

它才真正开始变成“你们的 agent”。

## 核心概念

### 1. 通用能力不等于项目理解

一个模型可以知道 React、TypeScript、Node.js 的通用知识，但这并不代表它知道：

- 当前项目的状态管理选型
- 哪些目录是公共层，哪些是业务层
- 测试应该放在哪里
- 哪些 API 约定已经被团队固定下来

所以“会写”不等于“会按这个项目的方式写”。

### 2. 项目知识应该是可维护资源

不要把所有项目知识都直接堆进 system prompt。

更好的方式通常是：

- 结构化文档
- 明确的项目约定
- 可以按需加载的上下文资源

换句话说，项目知识不应该只是“写给人类看的长文档”，也应该是：

- agent 可读取
- 可维护
- 可更新
- 可复用

### 3. 项目知识的价值是降低“看起来合理”的错误

很多 agent 生成的代码单看没问题，但放进项目就不对。

项目知识的作用，就是让 agent 从“通用正确”提升到“仓库内正确”。

## 什么算项目知识

项目知识不一定非得是厚重的设计文档。

很多高价值知识其实非常朴素，比如：

- 目录结构说明
- 架构原则
- 代码风格
- 常用命令
- 测试规范
- PR 约定
- 常见问题排查手册

这些内容对人类开发者有帮助，对 agent 同样有帮助。

区别只在于：

- 人类可能会“顺手记住”
- agent 必须被明确提供

## 为什么不能全塞进 prompt

一个常见冲动是：

> 那我就把这些项目知识都放进 system prompt 不就好了？

问题在于，这会很快造成三个后果：

- prompt 变得非常重
- 大量知识在多数任务中根本用不上
- 不同知识之间会互相干扰

所以更合理的方式通常是：

- 提供高价值、结构化的项目知识资源
- 让 agent 在需要时读取

这和第 11 章讲的 filesystem context 是一致的。

本质上，项目知识也是一种“按需拉取的外部记忆”。

## 推荐知识类型

如果你准备让教程或真实项目继续往前走，可以优先整理这些内容：

- 目录结构说明
- 架构原则
- 代码风格
- 常用命令
- 测试规范
- PR 约定
- 常见问题排查手册

这类知识的特点是：

- 对很多任务重复有价值
- 变化频率没有代码本身那么高
- 非常适合被 agent 读取

## 推荐示例任务

比如：

> “为这个 React 页面新增一个按钮，沿用项目现有的数据获取与样式约定。”

这个任务能很好体现项目知识的重要性，因为它不只是“能写出按钮”，还要“写得像这个项目里的代码”。

这类任务最容易暴露：

- 没接入项目知识时，agent 看起来会写，但风格不贴合
- 接入项目知识后，agent 的结果会明显更像“仓库内延续”而不是“外来代码”

## 这一章和前面章节的关系

前面的章节解决的是：

- agent 能不能推进任务
- 能不能控制上下文
- 能不能在仓库里工作

这一章开始解决的是：

- agent 能不能像这个团队一样工作

这一步其实非常关键。

因为一个通用 coding agent 是否真的能落地，往往就在这里分水岭：

- 通用能力强，但不懂项目
- 与项目贴合，能稳定协作

## 为什么项目知识是“后半段”主题

这一章放在第 13 章，而不是更前面，是有原因的。

因为在教学顺序上，必须先让读者理解：

- agent 怎么工作
- coding task 怎么推进
- 仓库上下文怎么处理

当这些基础打好之后，读者才会真正明白：

- 为什么“项目知识接入”不是装饰，而是垂直化的起点

如果太早讲这章，读者容易把它理解成“多加一点文档”。

其实它更像：

- 从通用 agent 走向团队 agent 的转折点

## 本章最重要的设计取舍

这一章要避免两个极端：

- 一切都塞进 prompt，导致上下文膨胀
- 完全不提供项目知识，指望模型自己猜

更好的方向是：

- 提供高价值、可维护、按需可取的项目知识资源

这也是这一章最重要的思想。

不是“给更多知识”，而是：

- 给正确的知识
- 给结构化的知识
- 在需要时给

## 本章真正要让读者学会什么

读完这一章，读者应该真正理解：

- coding agent 的真正可用性，很大程度上取决于它是否理解当前项目
- 项目知识不是 prompt 装饰，而是垂直化的基础设施
- 与其堆很多通用知识，不如优先提供高价值的项目约定
- 项目知识越可维护，agent 越有机会真正进入团队协作流

## 本章结论

- coding agent 的真正可用性，很大程度上取决于它是否理解当前项目。
- 项目知识应该外化为可维护资源，而不是全部塞进 prompt。
- 接入项目知识后，agent 才有机会从“通用助手”变成“团队助手”。
