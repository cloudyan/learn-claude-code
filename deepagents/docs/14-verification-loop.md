# 14. 验证闭环：改完不算完，验证通过才算完

## 这一章解决什么问题

这是整套教程里最接近“交付标准”的一章。

因为 coding agent 和普通内容生成 agent 的关键区别之一就在于：

- coding agent 的产出可以被环境验证

这件事会直接改变我们对“完成”的定义。

在普通文本任务里，“写完了”往往就意味着结束。  
但在 coding task 里，“改完了”往往只意味着：

- 第一阶段刚结束

这一章要解决的，就是这个关键差别。

## 为什么验证是核心能力

在工程任务里，“看起来对”远远不够。

真正重要的是：

- 能不能通过类型检查
- 能不能通过 lint
- 能不能通过测试
- 能不能在必要时完成构建

只有把这些环节纳入默认流程，agent 才可能真正可托付。

换句话说，验证不是额外附赠，而是 coding agent 成为“工程执行者”的必要条件。

## 先记住一句话

**没有验证闭环的 coding agent，本质上仍然只是一个自动编辑器。**

这句话可能有点重，但它很准确。

因为如果 agent 只是：

- 读文件
- 改文件
- 输出一段总结

那它当然已经比普通聊天机器人更进一步了，但它仍然没有完成工程任务里最关键的一步：

- 证明修改确实成立

所以验证闭环的意义是：

- 把“我觉得改对了”
- 变成“环境证明它至少在当前约束下成立”

## 核心概念

### 1. 修改必须回到环境里验证

模型不能凭语言感觉断言“这段代码应该没问题”。

真正可靠的方式是：

- 改动后执行验证
- 基于验证结果继续修复
- 验证通过后再结束

这也是 coding agent 与普通内容生成 agent 的根本差异之一。

### 2. 验证不止是 test

常见验证方式包括：

- `typecheck`
- `lint`
- `test`
- `build`
- `smoke test`

不同任务的验证标准不同，但验证闭环本身是通用原则。

所以这一章不是要告诉读者：

- “永远都跑全部验证”

而是要建立一个更准确的判断：

- “当前任务最相关的验证集合是什么”

### 3. 不是每次失败都要无限修复

验证失败后，agent 需要判断：

- 这是局部错误，适合继续修
- 还是已经超出当前任务边界，应该停止并上报

这也是可控性的一部分。

因为真正的工程系统不能只会：

- 一直试

它还必须会：

- 何时停止
- 何时升级给人类
- 何时报出部分完成状态

## 为什么验证闭环改变了任务结构

一旦把验证真正纳入默认流程，coding task 的结构就会从：

1. 理解任务
2. 修改代码
3. 汇报结果

变成：

1. 理解任务
2. 修改代码
3. 运行验证
4. 根据验证继续修复或结束
5. 汇报结果

这会让 agent 的主循环变得更接近真实工程工作流。

也正因为如此，验证闭环不只是一个“补充能力”，而是会改变整个任务推进方式。

## 推荐示例任务

比如：

> “修复一个类型错误，并确保 `pnpm typecheck` 和相关测试通过。”

这类任务最能体现验证闭环的价值。

因为你能很直观地看到：

- 修改只是中间阶段
- `typecheck` 和测试结果才决定任务能不能收口

## 为什么“改完就停”很危险

如果 agent 改完文件后就结束，常见风险包括：

- 修改引入了新的类型错误
- 改动破坏了已有测试
- 局部正确但全局不兼容
- 代码风格、构建要求没有满足

这些问题单靠模型直觉几乎不可能稳定发现。

所以这一章必须非常明确地建立一个标准：

**完成标准不应该是“文件已经改了”，而应该是“相关验证已经通过”。**

## 验证为什么也影响用户信任

验证闭环不只是工程正确性问题，也直接影响用户体验。

因为当 agent 最终交付时，如果它能明确说：

- 我改了什么
- 我跑了什么验证
- 哪些通过了
- 哪些没有继续处理

用户会明显更容易建立信任。

反过来，如果 agent 只是说：

- “我已经帮你改好了”

那用户很难判断：

- 它到底是改完了
- 还是只是生成了一版看起来不错的代码

## 如何选择验证范围

本章还需要帮助读者建立一个现实判断：

- 验证不是越多越好

如果每一个小任务都跑全量构建、全量测试、全量集成验证，成本会很快变得不可接受。

所以更合理的思路通常是：

- 优先跑与当前改动最相关的验证
- 必要时再升级到更大范围验证

例如：

- 类型相关改动先跑 `typecheck`
- 样式小改动先跑 `lint` 和轻量 smoke
- 核心逻辑改动再跑更完整测试

## 这一章和前面章节的关系

前面的章节已经建立了：

- 主线任务推进
- 文件系统上下文
- 上下文隔离
- 项目知识接入

但直到验证闭环接上之前，这套系统都还没有真正形成“交付标准”。

所以第 14 章很重要，因为它把前面的能力统一拉到了一个更实际的问题上：

- 什么时候才算真的完成

## 这一章和第 15 章的关系

第 14 章讲的是：

- 正确性交付

第 15 章讲的是：

- 可控性交付

这两章是连续的。

因为一个能验证的 agent，如果不可观察、不可中断、不可审批，也很难真正进入团队使用。

所以验证闭环和可控性，是工程化阶段最重要的两步。

## 本章最重要的设计取舍

这一章要强调：

- 没有验证的 coding agent，只能算自动编辑器
- 有验证闭环的 coding agent，才有机会成为工程执行者

同时也要提醒：

- 验证不应无限扩张
- 应优先选择与任务最相关的验证集合

也就是说，我们追求的不是“跑得最多”，而是：

- 跑得合适
- 跑得可解释
- 跑得能支撑交付

## 本章真正要让读者学会什么

读完这一章，读者应该真正理解：

- “改完就停”不是 coding task 的完成标准
- 验证结果应该成为 agent 下一轮决策的输入
- `typecheck`、`lint`、`test`、`build` 是不同层级的验证手段
- 验证闭环既提高工程正确性，也提高用户信任

## 本章结论

- “改完就停”不是 coding agent 的完成标准。
- 验证闭环是 coding agent 可托付性的核心来源之一。
- agent 应该把验证结果视为下一轮决策输入，而不是附加步骤。
- 当验证真正成为默认流程时，coding agent 才开始接近真实工程执行者。

## 下一章预告

下一章我们会进入另一个非常实际的问题：即使 agent 能工作、能验证，它是否足够可控、可观察、可中断。
