# 17. 做你自己的垂直 Coding Agent

- 不是“你已经学完了”

## 为什么要做垂直 Agent

通用 coding agent 很有价值，但在实际团队里，更容易落地的通常是“任务边界更清晰”的垂直 agent。

因为它们通常拥有：

- 更清晰的目标
- 更稳定的知识范围
- 更可控的验证标准
- 更低的失败成本

换句话说：

- 通用 agent 很有吸引力
- 但垂直 agent 更容易做成真正可用的工具

## 先记住一句话

**不要把“更通用”误解成“更好”。**

在很多团队场景里，更有价值的往往是：

- 更可控
- 更稳定
- 更容易评估
- 更容易嵌入现有工作流

所以教程最后一章选择收束到“垂直 agent”，而不是“更宏大的万能系统”，是一个很刻意的决定。

## 什么是垂直 coding agent

所谓垂直 coding agent，不是说它只能做一件事，而是说它的任务边界更清楚。

例如：

- 专做前端页面修复
- 专做测试修复
- 专做 code review
- 专做文档同步

这类 agent 的共同特点是：

- 输入更稳定
- 输出更明确
- 验证方式更清晰
- 风险边界更容易控制

这也是为什么它们通常更适合作为真实落地的第一步。

## 典型方向

### 1. 前端修复 Agent

适合的任务：

- 小范围页面修复
- 样式问题修复
- 类型错误修复
- 组件重构

这类 agent 的优势在于：

- 问题形态相对集中
- 视觉和类型反馈比较明确
- 容易建立稳定验证方式

### 2. 测试修复 Agent

适合的任务：

- 修复失败测试
- 补最小测试
- 更新快照

这是一个非常适合落地的方向，因为：

- 任务边界清晰
- 验证闭环天然明确
- 很容易做 benchmark

### 3. 文档维护 Agent

适合的任务：

- 更新 README
- 补 API 文档
- 根据代码变更同步说明

这个方向的价值在于：

- 风险较低
- 成本较低
- 很容易体现项目知识接入的价值

### 4. Code Review Agent

适合的任务：

- 审查风险点
- 检查遗漏测试
- 发现潜在回归

这类 agent 和“自动写代码”不同，它更强调：

- 判断
- 风险提示
- 结构化反馈

### 5. PR 辅助 Agent

适合的任务：

- 汇总改动
- 生成 PR 描述
- 检查发布说明

它的特点是：

- 很贴近团队真实工作流
- 风险较低
- 很适合作为 agent 能力切入协作流程的入口

## 如何从通用 Agent 走向垂直 Agent

可以沿着这条路径演进：

1. 先限定任务边界
2. 再限定工具集合
3. 再接入项目知识
4. 最后优化验证和交付格式

这通常比一开始就做“万能 agent”更容易成功。

因为一旦任务边界足够明确，你就更容易回答这些现实问题：

- 它到底该做什么
- 它不该做什么
- 怎么验证它是否有用
- 怎么控制它的风险

## 为什么不要一开始就做“万能 agent”

“万能 agent”最容易让人兴奋，但它也最容易失败。

因为它通常会带来：

- 目标不清
- 工具边界失控
- 验证标准模糊
- 评估困难

而垂直 agent 的优势恰恰是反过来的：

- 目标清晰
- 工具更少
- 验证更明确
- 失败成本更低

所以如果你真的想从教程走向项目，最好的下一步通常不是：

- “继续加更多抽象”

而是：

- “先挑一个真实且边界清晰的方向做起来”

## 如何判断一个垂直方向值不值得做

你可以用下面几个问题来判断：

- 这个任务是否经常出现
- 这个任务边界是否清楚
- 这个任务是否有明确验证方式
- 这个任务失败成本是否可控
- 这个任务是否能快速看到价值

如果一个方向在这些问题上的答案都比较明确，它通常就是很好的起点。
