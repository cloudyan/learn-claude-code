# 11. Filesystem Context：让 Agent 真正进入代码库

## 这一章解决什么问题

到了这里，我们已经有了：

- 最小可用 coding agent
- `Deep Agents` 主线入口
- planning 机制

但还有一个非常关键的问题：

**agent 到底怎么“理解”代码库。**

如果这个问题处理不好，agent 就算有再多工具，也很容易变成下面这种状态：

- 知道要做什么
- 但不知道从哪里开始找
- 或者读了太多无关内容
- 最后被上下文噪音淹没

所以这一章的重点是：

**为什么文件系统不是附属工具，而是 coding agent 的核心上下文来源。**

## 为什么消息历史不够

很多人初学 agent 时，会自然地把所有上下文都想象成“消息历史”。

这种想法在简单对话场景里没问题，但在 coding task 里很快就会崩。

因为一个真实代码仓库里会有：

- 成百上千个文件
- 很长的源码
- 很多局部相关、全局不相关的信息

如果试图把这些信息直接塞进消息历史，会马上出现几个问题：

- token 成本飙升
- 模型注意力被稀释
- 重要信息和无关信息混在一起
- 每走一步都越来越重

也就是说，消息历史适合保存：

- 当前任务状态
- 最近观察结论
- 上一步关键结果

但不适合承担“整个代码库”的角色。

## 先记住一句话

**代码库应该被视为外部记忆，而不是聊天上下文的延长线。**

这句话很重要，因为它会直接改变我们设计 agent 的方式。

如果你把代码库理解成“外部记忆”，那 agent 的正确工作方式就会变成：

- 需要时去找
- 找到后读局部
- 基于局部继续缩小范围
- 逐步建立任务相关理解

而不是：

- 一开始就把一大堆文件塞进上下文

## 为什么文件系统是核心上下文

对 coding agent 来说，代码库本身就是最重要的外部记忆。

模型不需要一开始就知道所有文件内容，它需要的是：

- 知道如何按需读取
- 知道如何缩小搜索范围
- 知道如何围绕工作区逐步建立局部理解

这意味着一个成熟一点的 coding agent，强大之处往往不在于：

- “它一次记住了多少内容”

而在于：

- “它有没有按合理顺序读取最相关的内容”

## 核心概念

### 1. 消息历史不是代码库的替代品

消息历史适合保存：

- 最近任务状态
- 当前结论
- 上一步输出

但不适合长期承载大量源码内容。

所以在 coding agent 里，消息历史更像：

- 任务推进日志

而代码库才是：

- 真正的知识来源

### 2. 文件系统是按需拉取的上下文来源

agent 应该在需要时：

- 列目录
- 搜索文本
- 读取相关文件
- 只把局部内容放进推理上下文

这是一种非常重要的工作方式变化。

它意味着：

- 理解不是一次性灌进去的
- 而是逐步搜索、逐步聚焦的

### 3. 工作区边界必须明确

coding agent 必须清楚：

- 当前 workspace root 是哪里
- 哪些路径可访问
- 哪些路径禁止修改

这既是安全问题，也是稳定性问题。

如果边界不明确，agent 会出现两种风险：

- 读到不该读的东西
- 改到不该改的地方

## 这一章真正关心的不是“读文件”，而是“读得有策略”

初学者很容易把这章理解成：

> 哦，就是给 agent 加 `read_file`

但那只是表层。

这章真正讨论的是：

- 先读什么
- 为什么先搜索再读
- 为什么不能全量灌入
- 为什么输出需要裁剪
- 为什么路径必须有边界

也就是说，我们讨论的是：

**仓库上下文获取策略。**

## 本章应覆盖的工程点

这一章最值得强调的工程点有：

- workspace root
- safe path
- output truncation
- large file handling
- 文本搜索优于全量读取

这些点看起来很“工程”，但它们本质上都服务于一件事：

让 agent 能在真实仓库里稳定建立局部理解。

## 为什么“先搜索再读取”是更好的默认策略

面对代码仓库时，一个常见错误是：

- 一上来就直接读某个大文件

更好的默认顺序通常是：

1. 先列目录，理解结构
2. 再搜索关键词，缩小范围
3. 再读最相关的文件片段
4. 必要时继续深入

这套顺序的好处是：

- 更便宜
- 更稳
- 更符合模型的注意力使用方式

所以这一章要明确传达一个观念：

**搜索不是辅助动作，而是理解仓库的第一步。**

## 推荐示例任务

比如：

> “找出认证模块的入口、核心 service 和测试文件，然后修复一个已知 bug。”

这个任务能很好体现：

- 为什么要先搜索
- 为什么不能一次把所有文件读进来
- 为什么代码库应该被视为外部记忆

它也是一个非常好的教学任务，因为它天然要求 agent：

- 定位结构
- 建立局部理解
- 再进入修改

## 为什么输出治理也属于 filesystem context 的一部分

很多人会把“输出裁剪”理解成一个纯工程优化问题。

其实它和这一章密切相关。

因为如果 agent 读取文件、执行搜索、运行命令后，把超长结果毫无节制地塞回上下文，那么你前面再好的“按需读取”策略也会被破坏。

所以这章要连着看：

- 如何读取
- 读取多少
- 如何裁剪
- 如何继续分页读取

这几件事本质上是一体的。

## 这一章和第 10 章的关系

第 10 章解决的是：

- 任务主线不要丢

第 11 章解决的是：

- 仓库上下文不要乱

这两章其实是互相补的。

如果只有 planning，没有 filesystem strategy，agent 会知道“要做什么”，却不知道“去哪里找”。

如果只有 filesystem strategy，没有 planning，agent 会很会读仓库，却不知道“为什么而读”。

所以它们组合起来，才构成真正稳定的任务推进基础。

## 这一章和第 12 章的关系

再往后，第 12 章会讲 subagents。

你会发现：

- planning 解决任务主线问题
- filesystem context 解决仓库上下文问题
- subagents 解决上下文隔离问题

这三章连在一起，才真正构成 `Deep Agents` 主线的中段核心。

## 本章最重要的设计取舍

这一章要明确传达一个观点：

- coding agent 的强大，很多时候不是因为它“记住得更多”
- 而是因为它“按需读取得更合理”

换句话说，优秀的 coding agent 不应该像：

- 把仓库尽量多塞进上下文的吞噬器

而应该像：

- 一个在仓库里有策略地搜索、读取、聚焦的工作者

## 本章真正要让读者学会什么

读完这一章，读者应该真正理解：

- 文件系统不是辅助工具，而是 coding agent 的核心上下文来源
- 代码库应该被视为外部记忆，而不是全部塞进消息历史
- 搜索通常应该先于阅读
- workspace 边界、路径安全、输出治理都属于“上下文策略”而不是纯工程细节

## 本章结论

- 文件系统不是附属能力，而是 coding agent 的核心上下文入口。
- 代码仓库应被视为外部记忆，按需读取比全量灌入更合理。
- workspace 边界、路径安全和输出治理，是这一层稳定性的核心。
- 从这一章开始，读者应该逐步建立“搜索优先、局部读取、逐步聚焦”的仓库工作方式。
