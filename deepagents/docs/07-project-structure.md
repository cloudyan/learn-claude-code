# 07. TypeScript 工程化组织

## 这一章解决什么问题

前面几章讨论的是能力和概念，这一章开始讨论工程形态：

- 代码应该放在哪里
- 模型和 provider 如何管理
- 工具如何拆分
- 哪些实现应该共享，哪些应该按章节独立

这章放在第 7 章的位置，不是偶然。

因为到这里为止，读者已经经历了两个明显不同的阶段：

- 第 3 章：最小 agent，单文件示例最有教学价值
- 第 6 章：最小 coding agent，能力开始组合，单文件已经不再合适

所以这一章的任务就是把这个变化讲清楚：

**为什么教程从这里开始，要从“单文件 demo”进入“可维护项目结构”。**

## 为什么项目结构很重要

agent 教程很容易写成“一章一个单文件 demo”，短期看很直观，长期看却有两个问题：

- 代码复用困难
- 读者很难把示例迁移到真实项目

也就是说，单文件 demo 非常适合：

- 快速理解最小闭环

但不太适合：

- 承载越来越多的共享能力
- 承载多章节的连续实现
- 支撑后续真实项目迁移

所以从这一章开始，我们要把教程从 demo 思维转向项目思维。

## 先记住一句话

**示例负责教学，`src/` 负责沉淀。**

这几乎可以作为本章的总纲。

如果你把所有东西都堆在示例里，教程会越来越难维护。
如果你把所有东西都提前抽进 `src/`，教程又会失去可读性。

所以更合理的方式是：

- `examples/` 负责“让读者看懂这一章”
- `src/` 负责“把会复用的能力沉淀下来”

这就是当前教程结构的核心原则。

## 为什么第 3 章不拆，但第 6 章之后开始拆

这是很多读者看到代码结构后最自然的问题。

第 3 章里，我们故意让示例保持很薄、很直白，因为那一章的重点是：

- 看懂最小 agent 闭环

如果一开始就把东西拆成：

- `providers/`
- `tools/`
- `prompts/`
- `agents/`

读者会更难第一眼看清：

- 模型在哪
- 工具在哪
- agent 怎么创建
- 调用入口在哪

所以第 3 章的“单文件”是一种教学取舍，不是工程终态。

而到了第 6 章之后，情况变了。

因为现在开始引入的东西已经越来越多：

- 文件工具
- 搜索工具
- planning
- prompt 分层
- `LangChain v1` agent 封装
- `Deep Agents` 封装

如果这时候还坚持所有东西都写在一个文件里，读者会越来越难看清边界。

所以从第 6 章开始分层，是合理的。

## 当前教程的推荐目录

```text
deepagents/
  docs/
  examples/
  src/
    providers/
    tools/
    prompts/
    agents/
    deep-agents/
    shared/
```

这个结构不是为了“显得像正式项目”，而是因为当前教程已经进入了一个需要分层的阶段。

## 三层结构分别解决什么问题

### 1. docs/

`docs/` 解决的是：

- 怎么把知识讲清楚
- 每章想让读者理解什么
- 章节之间如何承接

它的职责是叙事，不是运行。

### 2. examples/

`examples/` 解决的是：

- 每章对应什么可运行示例
- 读者怎样最快地把本章概念跑起来

它的职责是教学，不是长期复用。

也正因为如此，示例应该尽量：

- 薄
- 聚焦
- 一章只引入一个新的复杂度

### 3. src/

`src/` 解决的是：

- 哪些能力会在多章中复用
- 哪些实现应该被统一维护
- 哪些东西已经值得从“示例代码”沉淀成“共享代码”

它的职责是复用和沉淀。

## 目录职责

### providers/

负责模型初始化和 provider 适配，比如：

- OpenAI 兼容接口
- 模型名称与基础配置
- 环境变量读取

这里的核心思想是：

- 模型配置不要散落在每个示例里重复写

### tools/

负责工具定义与实现，比如：

- 文件读取
- 文件写入
- 文本搜索
- shell 执行
- planning 工具

这里最重要的原则是：

- 工具尽量原子化
- 工具命名和边界要稳定

而且在当前教程里，这一层也帮助我们明确了：

- 哪些工具属于 `LangChain v1` 路线
- 哪些工具在 `Deep Agents` 路线要避开内置保留名

### prompts/

负责系统提示和任务模板，比如：

- 通用 agent prompt
- coding agent prompt
- researcher prompt
- reviewer prompt

这一层的职责不是“做复杂模板系统”，而是：

- 不让 prompt 散落在每个示例脚本里

### agents/

承载基于 `LangChain v1` 的基础 agent 封装。

这层存在的意义是：

- 把“基础 agent 主线”稳定下来
- 让前半段教程有一致的代码落点

### deep-agents/

承载基于 `Deep Agents` 的主线 agent 封装。

这层的意义是：

- 把第 9 章之后的主线实现集中起来
- 明确和 `agents/` 是不同层级的承载方式

### shared/

承载公共能力，比如：

- safe path
- 输出裁剪
- 工作区定位
- 环境变量加载
- 日志工具

这层经常会被低估，但它实际上是教程工程从“示例堆积”走向“统一维护”的关键。

## examples/ 的角色

`examples/` 不是为了放所有共享逻辑，而是为了放每一章的可运行示例。

建议规则：

- 每个示例都独立可运行
- 每个示例只引入一个新的复杂度
- 示例尽量薄，共享逻辑下沉到 `src/`

这是当前教程最重要的工程边界之一。

如果示例变得太厚，就会出现两个问题：

- 读者一眼看不清本章重点
- 同样的实现逻辑会在多个示例里重复

## 为什么 `Deep Agents` 路线更需要复用 `src/`

到了 `Deep Agents` 主线之后，教程的关注点已经不再是：

- “一个文件里怎么把 agent 组装起来”

而是：

- “更高层抽象下，哪些能力已经值得被沉淀成共享实现”

比如：

- provider 配置
- prompt 规则
- Deep Agent 封装
- 工具命名约束

这些东西如果还继续散落在示例脚本里，教程会越来越乱。

所以你会看到：

- 第 9 章示例入口很薄
- 真正逻辑更多下沉到 `src/deep-agents/`

这不是“过度工程化”，而是因为主线复杂度已经到了值得复用的阶段。

## 目录如何服务章节主线

你可以把当前目录和主线章节这样对应起来：

- 第 3 章：`examples/03-first-agent`
- 第 6 章：`examples/06-minimum-capabilities` + `src/agents`
- 第 9 章：`examples/09-first-deep-coding-agent` + `src/deep-agents`
- 第 10-12 章：继续在 `src/` 上叠加 planning、filesystem、subagents 的解释

也就是说，目录结构不是独立存在的，它本身就在配合教程叙事。

## 为什么这里不直接设计成“企业级大仓库”

这一章的核心取舍是：

- 不把所有东西都写成“章节内局部脚本”
- 也不把教程一开始就设计成企业级大仓库

我们要找的是一个中间形态：

- 方便教学
- 也方便后续扩展

如果结构太简单，教程后面会乱。
如果结构太重，教程前面会难。

所以当前这套组织方式其实是一个很刻意的平衡：

- 前期可读
- 中期可扩
- 后期可落地

## 这一章和后续章节的关系

第 7 章的位置其实非常关键。

因为它刚好处在：

- 前半段基础 agent 理解
- 后半段 `Deep Agents` 主线展开

之间。

它的作用就是告诉读者：

- 从这里开始，教程不仅是在讲能力，也在讲这些能力应该如何组织

所以这章不是附属文档，而是整套教程从“概念展示”走向“工程实现”的转折点。

## 本章真正要让读者学会什么

读完这一章，读者应该真正理解：

- 为什么第 3 章适合单文件，而第 6 章之后开始需要分层
- 为什么 `docs / examples / src` 三层必须分开
- 为什么示例负责教学，`src` 负责沉淀
- 为什么当前目录结构是为教程主线服务，而不是纯粹为了工程形式

## 本章结论

- 工程化组织会直接影响教程的可维护性和读者迁移成本。
- `docs/`、`examples/`、`src/` 三层分工是这套教程的基础。
- 从这一章之后，教程会逐步从“概念演示”进入“主线实现”。
