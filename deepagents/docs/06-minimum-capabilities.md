# 06. Coding Agent 的最小能力集

## 这一章解决什么问题

到了这里，我们已经做出了第一个最小 agent。

但那个 agent 还不是 coding agent。

它可以：

- 回答问题
- 调简单工具
- 展示最小 tool calling 闭环

可一旦任务变成“去代码仓库里完成一件事情”，问题就立刻变了。

这一章要解决的问题是：

**一个最小可用的 coding agent，到底必须具备哪些能力。**

这里的关键词是两个：

- 最小
- 可用

也就是说，我们不是要列出一个“未来也许会需要的所有能力清单”，而是要找出：

**如果少了它，这个 agent 就很难真正完成代码任务。**

## 为什么先讲能力，不先讲框架

很多人在进入 coding agent 主题时，会先问：

- 用 `LangChain v1` 怎么写
- 用 `Deep Agents` 怎么写
- 要不要直接上 `LangGraph`

但如果不先把能力边界定义清楚，这些问题很容易变成：

- 看到什么工具就加什么
- 看到什么能力就堆什么
- 做着做着就变成一个边界不清的“万能 agent”

所以这一章先不急着继续扩框架，而是先把问题定义清楚：

- coding agent 最少要能做什么
- 为什么这些能力是必需的
- 它们是如何形成一个任务闭环的

## 先记住一句话

如果第 3 章的最小 agent 是：

**模型 + 工具 + 最小闭环**

那么这一章的最小 coding agent 则是：

**模型 + 仓库相关工具 + 计划 + 验证 + 收尾**

差别不在“代码变多了”，而在于：

**任务的世界变了。**

第 3 章的 agent 处理的是简单任务。
第 6 章的 coding agent 处理的是代码仓库中的多步任务。

## 为什么普通最小 Agent 不够

我们可以直接用一个常见 coding task 来看：

> “帮我修复这个 TypeScript 类型错误，并确认 typecheck 通过。”

如果仍然只靠第 3 章那种最小 agent，它会缺什么？

- 不知道去哪里找文件
- 不知道怎么搜索相关代码
- 不知道怎么修改代码
- 不知道如何验证修改是否正确
- 不知道应该先做什么、后做什么

也就是说，普通最小 agent 的问题不是“它不能回答”，而是“它无法在工程环境中持续推进任务”。

## 最小能力模型

这一章推荐把最小 coding agent 拆成五类能力。

### 1. 读代码

至少需要：

- `read_file`
- `list_files`
- `search_text`

为什么“读”是第一位？

因为 coding task 的第一步几乎永远不是改，而是理解。

你不能在没有看过仓库结构、没有看过目标文件、没有看过相关上下文的前提下稳定修改代码。

所以没有读能力，agent 就无法建立对代码库的基本认识。

### 2. 改代码

至少需要：

- `write_file`
- `edit_file`

只会读不会改，当然不是 coding agent。

但这里也要注意一个教学策略：

- “改代码”是最小能力之一
- 不等于第一时间就应该大量演示修改

因为修改一旦加入，复杂度会明显上升：

- 改错的风险
- 验证的必要性
- 对现有代码的影响

所以本章虽然把“改代码”列为最小能力的一部分，但配套示例仍然优先从低风险任务开始。

### 3. 运行验证

至少需要：

- `bash`

这里 `bash` 的价值不是“执行任何命令都很酷”，而是：

**让 agent 把修改重新放回工程环境里验证。**

比如：

- `pnpm typecheck`
- `pnpm test`
- `pnpm lint`

没有验证，agent 最多只能算“自动编辑器”；有了验证，它才开始接近“工程执行者”。

### 4. 维护计划

至少需要某种显式计划结构，比如：

- todo list
- plan steps

为什么 planning 在 coding task 中这么关键？

因为代码任务天然是多步的。

哪怕是一个很小的 bug 修复，通常也包含：

1. 找文件
2. 看实现
3. 判断改法
4. 修改代码
5. 跑验证
6. 汇总结果

如果这些步骤完全只存在模型“脑子里”，任务很容易漂移。

### 5. 汇报与收尾

agent 不只是完成任务，还要：

- 说明改了什么
- 说明为什么改
- 说明如何验证
- 说明是否还有残余风险

这一步非常容易被忽略，但它其实决定了 agent 能不能进入真正的团队协作场景。

因为工程里真正重要的，不只是“做了”，还包括“别人能不能接住你做的事”。

## 一个最小任务闭环

如果把这些能力放在一起，一个最小 coding agent 的闭环大概是：

1. 理解任务
2. 读取相关代码
3. 形成计划
4. 修改代码
5. 运行验证
6. 根据结果继续修复或结束
7. 汇报结果

你会发现，这比第 3 章的最小 agent 多了不少东西。

这正说明：

- coding agent 不是普通 agent 的“再加几个工具”
- 而是进入了另一种任务形态

## 配套示例

本章配套示例目录：

- [examples/06-minimum-capabilities](/deepagents/examples/06-minimum-capabilities/README.md)

和第 3 章不同，这一章的配套示例不再把所有东西都塞在单文件里，而是开始复用共享 `src/`。

这代表教程从这里开始进入一个新的阶段：

- 第 3 章强调最小闭环，所以示例是单文件
- 第 6 章开始强调“能力组合”，所以示例开始接入共享 `src/`

也就是说，读者在这一章第一次看到：

- prompt 不再写死在单个示例文件里
- 工具开始以模块形式复用
- coding agent 已经具备基本工程形态

## 配套代码如何对应能力模型

第 6 章的配套示例不是直接手写所有工具，而是复用共享源码中的最小 coding agent 结构：

- [create-coding-agent.ts](/deepagents/src/agents/create-coding-agent.ts)
- [tools/index.ts](/deepagents/src/tools/index.ts)

这里最重要的教学变化是：

- 我们不再只是讲“最小闭环”
- 而是在讲“最小能力组合”

这会自然把读者带向后面的工程分层章节。

## 当前示例覆盖了哪些能力

配套示例当前重点覆盖：

- `list_files`
- `search_text`
- `read_file`
- `bash`
- `plan`

这不是因为 `write_file` 和 `edit_file` 不重要，而是因为：

这一章的重点是让读者理解“最小 coding agent 闭环”，而不是一开始就演示大规模真实修改。

所以即便共享工具里已经有 `write_file` 和 `edit_file`，本章示例也更推荐先从：

- 搜索
- 阅读
- 形成计划
- 运行轻量命令

这样的低风险任务开始。

## 为什么本章示例先不改文件

这点非常重要。

很多人看到“coding agent”四个字，会自然期待示例立刻开始改代码。

但从教学顺序上，这并不总是最好的第一步。

因为一旦开始改文件，复杂度会同时提升：

- 要决定改哪里
- 要决定改多少
- 要决定怎么验证
- 要承担误改风险

而我们这一章真正想先建立的是：

**repo-aware 的工作方式。**

也就是：

- 先知道怎么进仓库
- 先知道怎么找文件
- 先知道怎么读上下文
- 先知道怎么形成最小计划

当这些东西先稳定下来，后面再进入“真实修改”，读者会更容易理解。

## 推荐运行方式

如果你已经在 `deepagents/.env` 中配置好：

- `OPENAI_API_KEY`
- `OPENAI_BASE_URL`
- `MODEL_NAME`

可以直接运行：

```bash
cd deepagents
npm run dev:03
```

默认任务是：

```text
阅读当前仓库的 deepagents/README.md，并给出一份执行计划，不修改文件
```

这个任务非常适合作为第一个最小 coding task，因为它已经具备这些特征：

- 目标在代码仓库里
- 需要读文件
- 需要形成计划
- 但风险很低

## 这个任务为什么设计得好

这个任务表面上很简单，但它已经能检验一个最小 coding agent 是否具备正确方向：

- 它会不会先看仓库
- 它会不会先读相关文件
- 它会不会形成有条理的执行计划
- 它会不会在不需要修改文件时保持克制

这其实是 coding agent 很重要的一种品质：

**不是一上来就动手，而是先理解任务边界。**

## 本章与前后章节的关系

和第 3 章相比，这一章新增的是：

- 面向代码仓库的工具集
- 最小 planning 能力
- 最小 repo-aware 工作方式

但这一章仍然没有完全进入 `Deep Agents` 主线。

它的作用是做一个桥：

- 前面是最小 agent
- 后面是更真实的 coding agent

它让读者自然过渡到第 8、9 章。

所以这章的位置非常关键。

如果没有这一步，教程会从“最小 agent”突然跳到“Deep Agents coding agent”，中间会缺少一个过渡层。

## 为什么这一章开始引入共享 src

这是教程结构上的一个转折点。

第 3 章故意不拆分，是为了先把最小结构讲透。

但从第 6 章开始，如果还坚持把所有东西都写在一个文件里，读者反而会更难看清：

- 哪些是模型层
- 哪些是工具层
- 哪些是 prompt
- 哪些是 agent 封装

所以这一章开始引入共享 `src/`，是合理的：

- 不是为了“更工程化而工程化”
- 而是因为能力开始多起来了，已经值得分层

## 本章最重要的设计取舍

这一章最重要的观点是：

- coding agent 的能力不应一开始就无限扩张
- 先把最小闭环做稳，再增加更高阶能力

所以本教程会坚持一个原则：

**每往前一章，只增加一个新的复杂度层次。**

第 3 章增加的是：

- 最小 agent 闭环

第 6 章增加的是：

- repo-aware 能力
- 最小 planning
- 最小验证意识

这样的推进方式，会比一开始就堆全部功能更稳。

## 本章真正要让读者学会什么

读完并跑完这一章，读者应该真正理解：

- coding agent 和普通 agent 的差别，不在“会不会调工具”，而在“能不能在工程环境中推进任务”
- 最小 coding agent 至少需要读、改、验证、计划和收尾五类能力
- 在教学顺序上，不必一上来就改文件，先建立 repo-aware 的工作方式更重要
- 从这一章开始，共享 `src/` 分层是合理的，因为能力组合已经变复杂了

## 本章结论

- 一个最小 coding agent 至少需要读、改、验证、计划和汇报五类能力。
- 这些能力共同构成了代码任务的执行闭环。
- 第 6 章的价值，不是让系统一下子变复杂，而是帮助读者从“最小 agent”过渡到“真正面向代码仓库工作的 agent”。
