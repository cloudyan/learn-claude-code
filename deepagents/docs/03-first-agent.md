# 03. 用 LangChain v1 写第一个最小 Agent

## 这一章解决什么问题

前两章讲清楚了心智模型和技术分层，这一章开始真正写代码。

但我们这一章的目标不是马上做一个“能读仓库、改代码、跑测试”的 coding agent，而是先把更基础的一件事讲清楚：

**一个最小 agent 到底是怎么工作的。**

换句话说，这一章要回答的是：

- `LangChain v1` 里的 `createAgent` 到底在做什么
- 工具调用到底是怎么和模型连接起来的
- 为什么一个很短的程序，也已经可以叫 agent

## 为什么从最小 Agent 开始

很多教程一上来就把下面这些东西全加进来：

- 文件系统
- shell 命令
- planning
- subagent
- memory
- verification

这样做当然可以很快搭出一个“更像样的 agent”，但也会让读者更难看清核心结构。

如果你在第一步就看到：

- 一堆工具
- 一堆 prompt
- 一堆工程分层

你很可能会觉得“系统能跑起来”，但不清楚它到底为什么能跑。

所以这一章故意只保留三样东西：

- 一个模型
- 两个工具
- 一个 agent

这就是最小闭环。

## 先记住一句话

这一章最重要的一句话是：

**Agent 不是更多代码，而是模型在工具环境里反复做决策。**

传统脚本的逻辑大多是：

```text
代码决定做什么 -> 调工具 -> 输出结果
```

而最小 agent 更接近：

```text
用户提出任务 -> 模型决定是否调工具 -> 工具返回结果 -> 模型继续决定 -> 输出结果
```

代码负责的是：

- 提供工具
- 声明约束
- 调用 agent

真正决定“下一步该做什么”的，是模型。

## 核心概念

### 1. createAgent 是 LangChain v1 的标准入口

这一章推荐直接使用 `createAgent`。

原因不是它最“高级”，而是它已经把最基础的 agent 形态稳定地表达出来了。

这能让我们把注意力放在真正重要的东西上：

- 模型
- 工具
- system prompt
- 消息输入

而不是过早陷入控制流细节。

### 2. 工具是模型和环境之间的桥

模型本身不会真的去读系统时间，也不会真的调用 shell，更不会自动执行 JavaScript 函数。

它能做的只是：

- 读工具说明
- 判断要不要使用工具
- 生成符合 schema 的参数

真正执行动作的，永远是工具实现本身。

所以工具可以理解成一座桥：

- 一边是模型的推理
- 一边是环境里的真实动作

### 3. 最小 agent 也有完整闭环

哪怕只有两个工具，一个最小 agent 也已经包含完整结构：

1. 接收用户任务
2. 让模型判断是否需要工具
3. 执行工具
4. 把结果返回给模型
5. 由模型组织最终回答

这就是最小闭环。

## 本章示例

本章配套示例目录：

- [examples/03-first-agent](/deepagents/examples/03-first-agent/README.md)

这个示例只用了两个工具：

- `echo`
- `get_time`

这两个工具都足够简单，所以你不会被业务逻辑干扰，能专注看清 agent 的工作方式。

## 为什么选这两个工具

`echo` 很适合做第一个工具，因为它几乎没有额外复杂度。

它可以帮助我们验证：

- 模型是否真的会调用工具
- 工具参数是否真的按 schema 传入
- 工具返回结果是否真的能被模型继续消费

`get_time` 的价值在于，它是一个模型自己不知道的“环境事实”。

模型不能可靠地自己猜“现在的时间”，但它可以：

- 判断这个问题需要查时间
- 调用工具
- 再根据工具结果完成回答

这一下就把“模型推理”和“环境动作”区分开了。

## 本章建议代码组织

```text
examples/03-first-agent/
  package.json
  tsconfig.json
  .env.example
  src/
    index.ts
```

这个示例故意保持为单文件入口，而不是一开始就拆成很多模块。

原因很简单：

- 第 3 章的目标是讲清最小 agent 闭环
- 不是讲工程化分层
- 如果一开始就把模型、工具、prompt 全拆出去，读者会更难一眼看清主流程

也就是说，这一章是“先看懂”，不是“先抽象”。

## 配套代码怎么读

在当前实现里：

- 模型直接在 `src/index.ts` 中初始化
- `echo` 和 `get_time` 直接在 `src/index.ts` 中定义
- `createAgent` 也直接在 `src/index.ts` 中调用

你可以按下面这个顺序去看代码。

### 第一步：先看模型初始化

这里最重要的不是 `ChatOpenAI` 这个类名，而是：

- agent 需要一个可调用的 chat model
- 模型名称、密钥、base URL 都来自环境变量

这意味着：

- 教程代码并不依赖某个特定厂商
- 只要底层服务兼容 OpenAI 接口，就能接进来

### 第二步：再看工具定义

这一步建议重点看三个东西：

- `name`
- `description`
- `schema`

它们分别决定：

- 模型怎么引用这个工具
- 模型怎么理解这个工具的用途
- 模型该如何构造参数

如果你忽略这三部分，工具再多也不稳定。

### 第三步：看 createAgent

`createAgent` 这一层做的事情，本质上是把下面这些东西组装起来：

- 模型
- 工具
- system prompt

在本章里，它没有承载特别复杂的状态管理，也没有复杂 middleware。

这正是它适合作为教程入口的地方。

### 第四步：看 agent.invoke

最后真正运行 agent 的地方，是：

- 给一个用户消息
- 调 agent
- 输出结果

这一步特别重要，因为它能帮助读者建立一个直觉：

**agent 不是“启动某个神秘系统”，而是像调用一个更强的 Runnable 一样调用。**

## 推荐运行方式

如果你已经在 `deepagents/.env` 中配置好：

- `OPENAI_API_KEY`
- `OPENAI_BASE_URL`
- `MODEL_NAME`

那么可以直接运行：

```bash
cd deepagents
npm run dev:03
```

它会使用默认任务：

```text
告诉我现在时间，并把这句话原样返回
```

如果你想传自己的任务，也可以这样运行：

```bash
npm run dev:03 -- "告诉我现在时间，并把这句话原样返回"
```

## 第一次运行时应该观察什么

第一次跑这个示例，我建议你不要急着关心“最终回答写得漂不漂亮”，而是先观察这两件事：

### 1. 模型有没有真的使用工具

如果 agent 没有调用 `get_time`，那说明：

- 要么 prompt 没写清楚
- 要么工具描述不够清楚
- 要么这个问题本身不够触发工具调用

### 2. 工具结果是怎么影响最终回答的

你要重点观察的是：

- 工具调用发生在中间
- 最终答案仍然是模型组织出来的自然语言

也就是说，工具不是最终回答本身，而是最终回答的依据之一。

## 一个最小任务是如何完成的

我们用这个任务举例：

> “告诉我现在时间，并把这句话原样返回。”

这个任务里，模型通常会经历下面的思考过程：

1. 用户要求两个结果
2. “现在时间”需要查工具
3. “原样返回”可以直接处理
4. 调 `get_time`
5. 拿到时间结果
6. 组织最终回答

你会发现：

- 任务并不复杂
- 但已经完整体现了“观察 -> 决策 -> 行动 -> 汇报”

这就是为什么它很适合作为第一个 agent 示例。

## 为什么这章不用 PromptTemplate

这一章的 prompt 是静态的，所以直接使用字符串常量就够了。

这里不需要 `PromptTemplate`，原因很简单：

- 没有复杂变量插槽
- 没有角色片段拼装
- 没有多版本模板切换

在教程前半段，优先追求可读性和直接性，比提前引入模板抽象更重要。

## 为什么这章不用共享 src 分层

你会注意到：

- 后面的章节会逐渐引入共享 `src/`
- 但这一章没有这么做

这不是偷懒，而是教学策略。

如果一上来就把东西拆成：

- `providers/`
- `tools/`
- `prompts/`
- `agents/`

读者虽然会觉得“很工程化”，但会更难一眼看清最小主流程。

所以这一章选择的是：

- 先把核心闭环讲透
- 后面再逐步引入工程分层

## 这一章故意不解决什么

这一章故意不解决下面这些问题：

- 如何读写文件
- 如何在仓库中工作
- 如何做 planning
- 如何做 verification
- 如何做 subagents

因为这些都属于 coding agent 的额外复杂度。

第 3 章只负责把“agent 的最小工作方式”讲透，为第 6 章的最小 coding agent 做准备。

## 本章要让读者真正学会什么

读完并跑完这一章，读者至少应该真正理解这几件事：

- `createAgent` 不是魔法，它只是把模型和工具组织成一个可调用 agent
- 工具不是模型的能力，而是模型可调度的外部动作
- 一个最小 agent 完全可以很短，但仍然具备完整闭环
- 模型回答和工具结果不是同一回事
- 工具设计的清晰程度会直接影响 agent 的稳定性

## 本章应强调的设计点

- 工具输入要清晰
- 工具输出要稳定
- 不要过早引入文件系统和 shell
- 目标是让读者看懂 agent 的最小工作方式
- 代码结构以“可读”为先，而不是以“抽象”优先

## 本章结论

- 最小 agent 的核心不是复杂，而是闭环。
- `LangChain v1` 的 `createAgent` 很适合作为教程第一步，因为它把模型、工具和调用方式表达得足够清楚。
- 这一章真正要建立的，不是“会写一个 demo”，而是“看懂 agent 的最小结构”。
