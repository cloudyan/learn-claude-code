# 01. Coding Agent 是什么

## 这一章解决什么问题

很多人第一次接触 agent 时，会把它理解成“会调用工具的聊天机器人”。

这个理解不算错，但对于 coding agent 来说远远不够。

这一章要解决的问题是：

- coding agent 到底和普通聊天机器人有什么不同
- coding agent 为什么不能只靠一轮问答完成任务
- coding agent 的最小组成部分是什么

如果这几个问题一开始不讲清楚，后面很容易出现一种误解：

- 好像只要模型能调几个工具，就已经是 coding agent 了

而这套教程想建立的不是这个认识。

## 为什么需要这个新视角

如果没有先建立正确心智模型，后面很容易把注意力放在框架 API 上，而忽略真正决定效果的部分：

- 工作区
- 工具边界
- 任务推进
- 验证闭环

对于代码任务来说，模型并不是只需要“回答问题”，而是需要在一个真实工程环境里持续感知、决策、执行、验证。

也就是说，coding agent 的问题不是：

- 它能不能说出一段正确的话

而是：

- 它能不能在代码环境里持续推进任务直到收口

## 先记住一句话

**Coding agent 不是更会聊天的模型，而是能在工程环境中持续工作的 agent。**

这句话几乎可以当作整套教程的起点。

因为只要把这个前提立住，后面很多设计选择都会变得清楚：

- 为什么需要文件系统
- 为什么需要 planning
- 为什么需要验证
- 为什么要关心权限和可控性

## 核心概念

### 1. Coding Agent 不是聊天机器人

聊天机器人主要处理：

- 一次输入
- 一次推理
- 一次输出

而 coding agent 面对的是：

- 读取代码库
- 分析结构
- 规划步骤
- 修改文件
- 执行测试
- 根据结果继续修复

它是一个多轮闭环系统，而不是一次性文本生成器。

所以这套教程后面才会反复强调：

- 不是输出有多漂亮
- 而是任务有没有真正推进

### 2. Coding Agent 的工作对象是工程环境

coding agent 不只面对自然语言，还面对：

- 文件系统
- 终端命令
- 测试框架
- 构建工具
- 类型检查器
- 团队约定

也就是说，它工作的“世界”不是聊天窗口，而是一个代码工作区。

这一点非常重要，因为它意味着：

- coding agent 的能力不是只发生在 prompt 里
- 它必须和环境持续交互

### 3. Coding Agent 的最小组成

一个最小 coding agent 至少需要：

- 模型：负责推理和决策
- 工具：负责执行动作
- 工作区：负责提供可读写的代码环境
- 计划机制：负责拆解复杂任务
- 验证机制：负责检查结果是否真的可交付
- 权限边界：负责避免错误操作扩大影响

这里最值得注意的是：

- “工具”并不是全部

如果把 coding agent 理解成“模型 + 工具”当然不完全错，但它会漏掉后面真正困难的部分：

- 任务结构
- 上下文治理
- 环境边界
- 验证和交付

## 最小心智模型

你可以把 coding agent 理解成下面这个循环：

```text
用户任务
  -> 模型理解问题
  -> 读取代码和环境
  -> 形成下一步动作
  -> 调用工具执行
  -> 观察结果
  -> 继续决策
  -> 验证结果
  -> 汇报完成
```

核心不是“会不会调工具”，而是：

**能不能在反馈中推进任务直到完成。**

这也是整套教程从第 3 章到第 14 章一直在反复强化的东西。

## 一个真实 coding task 为什么天然是多步的

比如一个很常见的任务：

> “帮我修复这个 TypeScript 类型错误，并补一个最小测试。”

这个任务通常至少包含：

1. 找出报错位置
2. 读取相关文件
3. 理解类型关系
4. 修改代码
5. 运行 typecheck
6. 补测试
7. 运行测试
8. 汇总结果

你会发现，这里面没有哪一步只是“继续聊天”。

它们都和真实工程环境发生关系。

所以 coding agent 本质上是一个任务执行系统，而不是简单的问答系统。

## 为什么“会调工具”仍然不够

这一点值得在第一章就提前埋下。

因为很多人一看到 tool calling，就会觉得：

- 好，agent 已经有手了

但真正的 coding agent 问题还包括：

- 什么时候该先规划
- 什么时候该先搜索
- 修改后如何验证
- 高风险动作如何控制

这些问题并不会因为“能调工具”就自动解决。

所以这一章的作用，就是先把问题空间定义出来。

## 这一章故意不急着写代码

这一章不急着写代码，是因为太早进入 API 会掩盖真正的结构问题。

如果我们现在直接上框架示例，读者可能会记住：

- 怎么 import
- 怎么写工具
- 怎么调用模型

但不一定会真正理解：

- 为什么 coding agent 需要文件系统
- 为什么要强调验证
- 为什么要处理权限边界

所以第一章先把“问题空间”定义清楚。

## 这一章和后续主线的关系

这一章最重要的作用，不是讲很多术语，而是给后面章节建立一个统一视角。

后面你会看到：

- 第 3 章开始讲最小 agent
- 第 6 章开始定义最小 coding agent 能力
- 第 9 章切入 `Deep Agents`
- 第 10-15 章逐步展开任务推进、上下文、验证和可控性

而这一切，其实都建立在第一章的一个前提上：

**coding agent 是在工程环境中工作的任务系统。**

## 本章真正要让读者学会什么

读完这一章，读者应该真正理解：

- coding agent 不是更会聊天的机器人
- 它的工作对象是代码仓库和工程环境
- 它必须处理任务推进、验证和权限边界
- “模型 + 工具”只是起点，不是全部

## 本章结论

- coding agent 不是会调工具的聊天机器人，而是在代码环境里持续工作的 agent。
- 它处理的是多步、带反馈、需要验证的工程任务。
- 构建 coding agent 不能只看模型调用，还要看工作区、工具、计划、验证和边界。
