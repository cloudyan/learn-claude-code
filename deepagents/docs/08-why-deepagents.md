# 08. 为什么进入 Deep Agents

## 这一章解决什么问题

到这里为止，我们已经：

- 讲清了 coding agent 的问题空间
- 建立了最小能力模型
- 用 `LangChain v1` 解释了基础 agent 结构

现在要回答的问题是：

- 为什么主线要从基础 agent 转向 `Deep Agents`

这一章的作用非常像一座桥。

它连接的是：

- 前面的“理解阶段”
- 后面的“主线实现阶段”

## 为什么不能一直停留在基础 Agent

基础 agent 很适合教学，但在真实 coding task 中，通常会逐渐暴露这些问题：

- 任务越复杂，越需要更强规划
- 代码库越大，越需要更稳定的文件系统工作方式
- 任务越长，越需要更好的上下文治理
- 一旦开始拆子任务，就需要更自然的 subagent 机制

也就是说，基础 agent 适合“讲清楚”，但不一定最适合“主线落地”。

这是一个很重要的教程判断。

因为如果我们一直停留在基础 agent 层，后面会发生两件事：

- 教程越来越像“手搓样板代码”
- 真正的 coding agent 主线反而越来越模糊

## 先记住一句话

**`Deep Agents` 不是为了更炫，而是为了让 coding agent 主线更自然。**

这句话很重要。

很多人看到更高层抽象时，第一反应会是：

- 是不是只是更封装一点

但真正的重点在于：

- 这个抽象是不是更贴近你要解决的问题

而 coding agent 的主线问题，从这里开始已经非常明确了：

- 文件系统
- planning
- subagents
- 长任务推进
- 上下文治理

这正是 `Deep Agents` 发挥价值的地方。

## 核心概念

### 1. Deep Agents 是更接近真实工作形态的 harness

这里最重要的不是“功能更多”，而是：

- 更贴近复杂任务
- 更贴近工作区环境
- 更贴近多步推进

也就是说，`Deep Agents` 更像是：

- coding agent 的主线工作平台

而不是一个只负责展示 API 的薄层封装。

### 2. 它不是替代 LangChain，而是建立在其之上

`Deep Agents` 不是另一套完全无关的体系。

在这套教程里，它的角色是：

- 延续前面的 agent 概念
- 提供更适合 coding task 的抽象

所以进入 `Deep Agents`，不代表前面 `LangChain v1` 的内容白学了。

恰恰相反，前面学的：

- tool calling
- schema
- 最小 agent 闭环

会让读者在这里更容易看懂：

- `Deep Agents` 到底帮你抽掉了哪些样板
- 它保留了哪些核心思想

### 3. 它也不是 LangGraph 的替代品

如果任务上升到明确状态机、审批节点、恢复执行等层次，仍然要考虑 `LangGraph`。

所以教程的分层依然成立：

- `LangChain v1`：理解基础
- `Deep Agents`：主线实现
- `LangGraph`：复杂 orchestration

这也是为什么第 8 章必须在第 9 章前面讲清楚。

不然读者很容易误解：

- 好像 `Deep Agents` 是最终一切问题的答案

其实它更准确的定位是：

- coding agent 主线最合适的中层抽象

## 从能力需求看为什么要切换

前面定义的最小能力中，以下几项在复杂任务里尤其关键：

- planning
- filesystem context
- subagents
- verification loop

这些能力如果全靠手搓，不是做不到，而是教程主线会很快被样板代码淹没。

`Deep Agents` 的价值就在于，它让我们能把注意力放在：

- agent 应该如何工作
- coding task 应该如何设计

而不是把大量篇幅花在重复性基础设施上。

这就是为什么本教程会在这里切换主线，而不是继续无限延长基础 agent 章节。

## 一个很实际的工程差异

当教程主线切到 `Deep Agents` 后，我们不应该继续机械地沿用前面手写 agent 的全部工具命名。

原因是 `Deep Agents` 自带一批内置工具和中间件能力。在当前本地验证的 `deepagents@1.8.8` 里，以下名字至少应视为保留名：

- `ls`
- `read_file`
- `write_file`
- `edit_file`
- `glob`
- `grep`
- `execute`
- `task`
- `write_todos`

这意味着：

- 前面 `LangChain v1` 示例里自定义的文件工具命名，在 `Deep Agents` 主线中不能原样照搬
- 对于文件读写、搜索、执行这类能力，应优先理解和复用 `Deep Agents` 的内置工具
- 我们只补充那些没有命名冲突、且确实有额外价值的自定义工具

这也是教程主线切换后一个很重要的真实工程信号：

**进入更高层抽象时，不只是 API 变了，工具边界和命名约束也会一起变化。**

## 为什么这章必须单独存在

如果没有第 8 章，教程会显得像是：

- 前面都在讲基础 agent
- 第 9 章突然直接换技术栈

读者会自然产生困惑：

- 为什么这里突然不用前面的方式了
- 是不是前面的东西已经不重要了

而第 8 章就是要把这个误解提前拆掉。

它要建立的是一个更准确的认识：

- 前面的内容是基础
- 第 8 章解释为什么基础之后应该切换主线承载层
- 第 9 章才是正式进入新的主战场

## 这一章和第 9 章的关系

第 8 章回答的是：

- 为什么要切换

第 9 章回答的是：

- 切换之后第一个可用版本怎么做

它们是非常标准的一对“桥接章 + 实战章”关系。

也正因为如此，第 8 章不需要堆太多实现细节。

它最重要的任务，是让读者在进入第 9 章前心里已经有一个稳定判断：

- 这里不是随便换工具
- 而是因为问题层级变了

## 本章最重要的设计取舍

这一章的取舍是教程定位上的取舍：

- 如果目标是“理解 agent 原理”，基础 `LangChain v1` 已经足够
- 如果目标是“做出一个可用的 coding agent”，教程主线应该切到 `Deep Agents`

这其实就是整套教程最重要的结构取舍之一。

因为它决定了：

- 你是想继续做“更复杂的基础 agent 教学”
- 还是开始真的做“可落地的 coding agent 主线”

## 本章真正要让读者学会什么

读完这一章，读者应该真正理解：

- `Deep Agents` 的价值不在“更高级”，而在“更贴近 coding agent 主线”
- 它不是替代 `LangChain`，而是建立在其上
- 它也不是 `LangGraph` 的替代品
- 技术栈切换背后真正变化的是：任务形态、工具边界和主线承载方式

## 本章结论

- `Deep Agents` 是这套教程进入真实 coding agent 主线的分界点。
- 它让我们能更自然地承载规划、文件系统、子任务和复杂任务推进。
- 这不是抛弃基础 agent，而是在理解基础之后进入更贴近实战的抽象层。
