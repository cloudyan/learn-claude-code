# 05. 为什么普通 Agent 做不好 Coding Task

## 这一章解决什么问题

到了这一章，读者已经知道：

- agent 是什么
- tool calling 是怎么回事
- 最小 agent 怎么工作

接下来要解决的问题是：

- 为什么这些能力放在一起，还是不够做 coding agent

这章非常关键，因为它是整套教程真正从“通用 agent”转向“coding agent”的分水岭。

## 为什么普通 Agent 会失效

普通 agent 往往能完成一些轻量任务，比如：

- 搜索信息
- 查文件
- 调简单 API

但一进入真实 coding task，就容易出现明显问题。

这不是因为模型突然变笨了，而是因为任务结构变了。

搜索信息和真正推进代码任务，本来就不是同一类问题。

## 先记住一句话

**普通 tool-calling agent 的问题，不只是工具不够，而是任务推进结构不够。**

这句话非常重要。

因为很多人一发现 agent 不行，第一反应会是：

- 那就再加几个工具

但真实情况往往是：

- 工具已经够多了
- 缺的是 planning、上下文治理、验证闭环和任务边界

所以这一章的重点是：

- 先把失败模式看清楚

## 核心失效模式

### 1. 任务漂移

coding task 往往包含多个步骤。

如果 agent 没有显式计划，它很容易在中途忘记原始目标，只围绕局部细节打转。

这类现象在真实任务里非常常见：

- 一开始是为了修一个 bug
- 后来开始纠结局部实现细节
- 再后来忘了还要补测试和做验证

这说明问题不在“会不会调工具”，而在“任务主线有没有被持续维持”。

### 2. 上下文污染

长任务中会累积大量无关中间结果。

如果没有有效的上下文治理，模型后续推理很容易被历史细节噪音拖累。

你会看到 agent 似乎什么都知道一点，但主线越来越模糊。

这也是为什么后面教程会专门讲：

- filesystem context
- subagents
- summarization

### 3. 不会验证

普通 agent 很容易在“改完代码”这一刻就认为任务结束。

但对于工程任务来说，真正的完成标准通常是：

- `typecheck` 通过
- `lint` 通过
- `test` 通过
- 必要时 `build` 通过

这一点是 coding agent 和普通生成式 agent 最大的差别之一。

### 4. 缺少 repo awareness

coding agent 不是在真空中写代码，而是在一个有目录结构、有约定、有依赖关系的代码库里工作。

如果 agent 不理解这些结构，就容易做出表面正确、全局错误的修改。

例如：

- 改对了当前文件
- 但破坏了整体架构约定
- 或把代码写到了错误的层级里

### 5. 权限边界不清

普通 agent 一旦拿到强工具，如果没有边界控制，很容易：

- 修改错误文件
- 执行不必要命令
- 输出过大结果
- 对工作区造成意外破坏

这类问题在 demo 里不明显，在真实项目里会立刻变成风险。

## 一个典型失败案例

比如任务是：

> “修复登录模块中的类型错误，补测试，并更新相关注释。”

普通 agent 可能会：

- 一开始就直接改代码
- 改完没跑 `typecheck`
- 完全没补测试
- 或者在测试失败后陷入局部循环

表面上它“会调工具”，但实质上它并没有形成可靠的任务推进能力。

这个例子特别好，因为它能说明：

- 问题不是会不会写一段代码
- 而是能不能把整个任务真正完成

## 为什么“会调工具”不足以构成 coding agent

这是整章最想建立的判断。

tool calling 解决的是：

- agent 能不能做动作

而 coding task 还要求：

- agent 能不能按正确顺序做动作
- agent 能不能知道什么时候停
- agent 能不能验证是否完成

所以这章本质上是在告诉读者：

**coding agent 的关键不在“动作能力”，而在“任务结构能力”。**

## 这一章和第 6 章的关系

这章做完之后，读者应该自然会问：

- 那如果普通 agent 不够，一个最小 coding agent 到底应该具备什么？

这正是第 6 章要回答的问题。

所以这章的作用是：

- 把“为什么不够”讲清楚

第 6 章的作用是：

这两章是成对出现的。

## 为什么这章必须在第 6 章前面

如果没有先讲这章，读者看到第 6 章列出的那些能力时，很容易觉得：

- 好像只是列了个清单

但只要先理解了这章的失败模式，就会很自然地明白：

- 为什么要有 planning
- 为什么要有 repo-aware 工具
- 为什么要有验证闭环
- 为什么要有权限和边界

所以这章不是“负面分析”，而是在为能力模型铺地基。

## 本章最重要的设计取舍

这一章的核心不是否定普通 agent，而是指出：

- 普通 tool-calling agent 是起点
- coding agent 是更高要求的工程形态

所以接下来我们要补的，不是“更多工具”，而是“更适合工程任务的能力结构”。

这就是整套教程后面切到：

- 最小 coding agent
- `Deep Agents`
- planning
- filesystem context
- verification

这一整条主线的真正理由。

## 本章真正要让读者学会什么

读完这一章，读者应该真正理解：

- 普通 agent 的问题不只是能力少，而是任务推进结构弱
- coding task 的核心难点在于主线、上下文、验证和边界
- 如果不先看清这些失败模式，后面的能力清单就会显得抽象

## 本章结论

- 普通 agent 的问题不只是工具不够，而是任务推进机制不够。
- coding task 需要规划、上下文治理、repo awareness 和验证闭环。
- 这也是为什么后续教程会逐步进入 `Deep Agents` 主线。
