# 10. Planning：让 Agent 先想清楚再动手

## 这一章解决什么问题

前一章我们已经有了一个最小可用的 `Deep Agents` coding agent。

但只要任务稍微复杂一些，新的问题就会马上出现：

- agent 会不会半路跑偏
- 会不会忘记还有哪些子任务没做
- 会不会改了一处之后漏掉验证或收尾
- 会不会在局部细节里打转，失去对整体任务的把握

这一章要解决的，就是 planning。

换句话说，这一章关注的不再是：

- agent 有没有能力

而是：

- agent 能不能把这些能力按正确顺序组织起来

## 为什么 planning 在这里变成主线问题

到了第 9 章，我们已经进入了 `Deep Agents` 主线。

这意味着问题开始从“怎么搭出最小闭环”升级成：

- 怎么让闭环在复杂任务里稳定工作

而 coding task 的一个根本特点，就是它很少是单步完成的。

哪怕是一个看起来很小的任务，也可能包含：

- 阅读相关代码
- 理解调用链
- 判断修改点
- 做局部修改
- 补测试
- 跑验证
- 汇总结果

如果这些步骤没有被显式化，模型很容易：

- 跳步
- 漏步
- 在局部细节中迷失
- 提前结束

所以 planning 在这里不是“可选增强”，而是主线里第一个真正决定任务稳定性的结构能力。

## 先记住一句话

**Planning 不是给用户看的列表，而是给 agent 的认知脚手架。**

很多人一提到 plan，会下意识想到：

- 看起来更工整
- 能展示任务步骤
- 更像“专业 agent”

这些都不是错，但都不够核心。

真正重要的是：

- 它帮助 agent 持续对齐目标
- 它帮助 agent 知道自己当前在第几步
- 它帮助 agent 在长任务中不丢主线

所以 planning 的本质不是“展示”，而是“对齐”。

## 为什么不能只相信模型“自己会记住”

一个常见误区是：

> 模型已经很强了，规划是不是可以完全留在它脑子里？

问题在于，coding task 里的上下文不是静态的。

随着任务推进，会不断出现：

- 新线索
- 新文件
- 新错误
- 新验证结果

这些信息会持续挤占上下文注意力。

如果没有显式 planning，模型虽然可能一开始知道目标是什么，但随着任务变长，目标会越来越容易被局部细节稀释。

所以 planning 的一个现实作用就是：

**把“主任务”从短暂推理，变成可持续对齐的外部结构。**

## 核心概念

### 1. 计划是认知脚手架

planning 的价值不只是“记录任务”，更重要的是给模型一个持续对齐目标的脚手架。

它让 agent 能在复杂任务中随时回答：

- 我现在在做什么
- 后面还有什么要做
- 这一步完成后应该切到哪里

### 2. 计划不应该太细

如果计划细到每个键盘动作，模型会被流程束缚。

比如这种计划就过细：

1. 打开文件
2. 找到第 20 行
3. 删除一行
4. 新增一行

这种粒度对人类操作说明有用，但对 agent 的任务推进帮助不大。

如果计划粗到只有一句“修复 bug”，它又起不到帮助作用。

所以 planning 的关键不是“有没有”，而是“粒度是否合适”。

### 3. 计划应该服务于执行

一个好的计划结构应该帮助 agent：

- 明确当前步骤
- 知道下一步做什么
- 在完成后更新状态

而不是变成纯展示性的装饰。

也就是说，planning 不是把任务写出来就结束了，而是要持续参与执行过程。

## 一个坏计划长什么样

坏计划通常会落入两个极端。

### 极端一：根本没有计划

这种情况下，agent 往往会：

- 一上来就读文件
- 一会儿改这里
- 一会儿改那里
- 最后忘了自己原本还要补测试或做验证

你会感觉它“做了很多事”，但任务推进不稳定。

### 极端二：计划过度流程化

这种情况下，agent 会被太细的流程束缚，像在执行一份僵硬剧本。

结果通常是：

- 对新线索反应变慢
- 面对真实情况变化时不灵活
- 计划本身比任务还重

所以真正好的 planning，在于：

- 足够有结构
- 但不过度僵化

## 推荐示例任务

本章适合用一个典型多步任务：

> “重构一个小模块，补一个测试，并更新相关注释。”

这个任务非常适合讲 planning，因为它天然包含多个目标：

- 改代码
- 补测试
- 改说明

如果没有计划，agent 很容易只完成其中一部分。

## 为什么 planning 应该放在 Deep Agents 主线早期

你可能会问：

为什么不先讲 filesystem，或者先讲 subagents？

原因是：

- filesystem 解决的是“如何更好读取上下文”
- subagents 解决的是“如何更好隔离上下文”
- planning 解决的是“任务主线怎么不丢”

而一旦任务主线会丢，后面再多上下文技巧都只是补救。

所以在主线展开里，planning 是更早应该被讲清楚的东西。

## 为什么 planning 不等于重型工作流

这里很容易出现另一个误区：

> 既然 planning 很重要，那是不是应该把 agent 做成一个严格流程引擎？

不一定。

这一章要特别提醒读者两件事：

- 不要把 planning 设计成重型流程引擎
- 也不要完全相信模型会“自动记住一切”

我们的目标是：

**给 agent 足够的结构，但不把它锁死。**

这也是为什么本教程主线到现在还没有直接下沉到 `LangGraph`。

我们当前仍然在强调：

- agent autonomy
- task structure
- 可持续推进

而不是先上显式状态机。

## 一个好的计划应该回答什么

对于 coding task，一个好的计划通常至少应该回答：

- 先看哪里
- 再改哪里
- 改完要验证什么
- 收尾还剩什么

你会发现，这种计划不是“操作说明书”，而更像：

- 任务导航图

这也是它最适合 agent 的地方。

## 和用户体验的关系

planning 不只是帮助 agent，对用户体验也有明显帮助。

因为显式计划能让用户更容易判断：

- agent 是否理解了任务
- agent 当前推进到哪一步
- 有没有漏掉关键目标

这会显著提升可控性。

所以 planning 同时服务于两端：

- 对 agent：帮助推进任务
- 对用户：帮助建立信任

## 需要强调的失败模式

本章可以对比两种行为：

- 没计划时，agent 容易漂移
- 有计划时，agent 能更稳定地推进任务

重点不是证明计划永远必要，而是说明：

复杂任务中，显式 planning 通常会显著提高可控性。

尤其是下面这类任务：

- 多目标任务
- 跨多个文件的任务
- 需要修改后再验证的任务
- 需要最终交付说明的任务

这些任务几乎都会从 planning 中明显受益。

## 本章最重要的设计取舍

这一章的关键取舍是：

- planning 要足够结构化
- 但不要重到把 agent 变成流程引擎

也就是说，我们要避免两个极端：

- 完全不做计划
- 一上来就把计划系统做成重型 orchestration

这也是为什么本教程的主线顺序是合理的：

- 先最小 agent
- 再最小 coding agent
- 再 Deep Agents 主线
- 再 planning

因为只有走到这里，读者才真正能感受到 planning 不是“锦上添花”，而是“避免主线丢失”的关键机制。

## 本章真正要让读者学会什么

读完这一章，读者应该真正理解：

- planning 的本质不是展示，而是认知脚手架
- coding task 越复杂，planning 的收益越明显
- 好计划的关键是粒度合适，而不是越细越好
- planning 是 `Deep Agents` 主线稳定推进的第一层结构能力

## 本章结论

- planning 是 coding task 中的重要认知脚手架。
- 计划的关键不在于复杂，而在于能帮助任务持续推进。
- 它既帮助 agent 对齐任务，也帮助用户理解 agent 当前状态。
- 后续章节里的 filesystem、subagents 和 verification，都会和 planning 紧密耦合。
