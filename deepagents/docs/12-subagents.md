# 12. Subagents：上下文隔离，不是多代理炫技

## 这一章解决什么问题

当任务变长、上下文变多时，即使有 planning 和 filesystem context，主代理仍然可能被大量局部细节拖慢。

这时一个自然的问题是：

- 要不要拆子代理

这一章就是专门回答这个问题的。

但我们会非常克制地回答它。

因为 subagents 是一个很容易被神化的话题，一不小心就会从“工程手段”变成“架构表演”。

## 为什么要谨慎讲 subagents

很多讨论会把 subagents 讲成：

- 同时跑很多 agent
- 自动分工协作
- agent swarm
- 大规模并行

这些方向当然不是没有价值，但对这套教程当前的主线来说，它们都还太早。

因为在 coding agent 的入门和主线实现阶段，subagent 最先要解决的，不是“规模”，而是：

**上下文隔离。**

这句话非常重要。

如果读者这一章只记住一件事，那应该是：

**subagent 的第一价值不是并行，而是把噪音隔离出去。**

## 为什么主代理会被上下文拖慢

一个真实 coding task 往往会产生很多“局部信息”。

比如：

- 某个目录结构的探索结果
- 某个模块依赖关系的分析
- 某个文件中间状态的观察
- 某次失败验证的具体日志

这些信息在局部阶段非常重要，但它们不一定都值得长期留在主上下文里。

如果所有探索过程都直接堆在主代理上下文中，常见结果是：

- 主线越来越模糊
- 后续决策越来越受噪音干扰
- 模型开始围着局部细节打转

所以 subagents 的根本价值是：

- 把局部复杂度隔离到局部上下文里处理

## 先记住一句话

**Task 不是为了显得聪明，而是为了保持主上下文干净。**

这句话几乎可以直接作为这一章的总纲。

如果一个子任务并不会显著污染主上下文，那就不一定值得拆 subagent。

反过来，如果一个子任务会产生大量中间噪音，那它就非常适合隔离出去。

## 核心概念

### 1. Subagent 最重要的作用是把噪音隔离出去

有些子任务会产生大量中间信息，比如：

- 探索目录结构
- 搜索调用链
- 分析某个模块的设计

如果这些信息全部留在主对话里，后续决策质量往往会下降。

而 subagent 可以把这种任务变成：

- 局部探索
- 局部总结
- 把高密度中间过程压缩成对主代理有价值的结论

### 2. 不同角色可以拥有不同职责

最常见的几种子代理角色可以是：

- `researcher`：负责找信息、读代码、不修改
- `coder`：负责局部实现
- `reviewer`：负责检查修改与风险

这类角色划分并不是为了做“组织图”，而是为了让任务边界更清晰。

### 3. Subagents 不等于必须并行

在本教程里，subagent 的重点不是并行度，而是任务边界和上下文边界。

先把“隔离”讲明白，比一上来做“多代理协作”更重要。

这是一个教学顺序问题，也是一个工程取舍问题。

## 什么情况下值得拆 subagent

最值得拆的情况通常有这几类：

### 1. 探索型任务

例如：

- “先帮我摸清这个模块的调用链”
- “先看一下认证逻辑散落在哪些文件里”

这类任务通常会产生很多中间信息，非常适合交给研究型子代理。

### 2. 审核型任务

例如：

- “帮我检查这次改动有没有回归风险”
- “看看这个修复有没有漏测”

这类任务需要不同的注意力分配方式，也适合单独角色承担。

### 3. 与主线目标不同步的局部任务

例如主代理现在在推进“修 bug”，但突然需要一次局部探索：

- “顺手查一下这个 util 还在哪些地方被引用”

这类任务本身不该把主代理主线打乱，适合下放。

## 什么情况下不值得拆

同样重要的是要知道，很多情况其实不需要 subagents。

例如：

- 任务很短
- 主上下文还很干净
- 子任务和主任务几乎完全同步
- 拆分后的沟通成本比收益更高

这时强行拆 subagent，反而会带来：

- 额外复杂度
- 更多协调成本
- 不必要的“多代理幻觉”

所以这一章很重要的一个判断标准是：

**只有在主上下文明显被噪音拖累时，才值得拆。**

## 推荐示例任务

比如：

> “先分析认证模块依赖关系，再修复登录流程中的类型错误，并检查测试覆盖。”

这个任务很适合拆成：

- `researcher` 先做局部探索
- `coder` 基于总结做修改
- `reviewer` 检查改动与验证结果

这个例子之所以好，是因为它能非常直观地展示：

- 为什么子代理不是为了“更炫”
- 而是为了让主流程更清晰

## 为什么角色要清晰

子代理一旦开始使用，就必须尽量让它们的职责清晰。

如果每个子代理都能：

- 探索
- 修改
- 审核
- 决策

那它们很快就会变成“另一个主代理”，而不是局部职责代理。

所以本章要强调一个很重要的设计原则：

**子代理不是更小的主代理，而是边界更清晰的局部角色。**

## 结果交接为什么重要

subagent 不是做完就完了，它还必须把结果交给主代理。

所以真正重要的不只是：

- 有没有 subagent

还包括：

- 它返回的总结是否清晰
- 它返回的是过程噪音，还是提炼后的结论

一个好的 subagent 输出应该更像：

- 可消费的局部结论

而不是：

- 再给主代理扔回一大堆原始日志

## 这一章和前两章的关系

你可以把这三章连起来看：

- 第 10 章：planning 解决任务主线问题
- 第 11 章：filesystem context 解决仓库理解问题
- 第 12 章：subagents 解决上下文隔离问题

它们分别对应 `Deep Agents` 主线中三个非常现实的痛点：

- 主线会不会丢
- 仓库会不会读乱
- 上下文会不会被噪音拖垮

所以这一章不是可有可无的花样，而是主线稳定性的第三块基石。

## 为什么本章不主打“多代理协作”

这套教程在这一章刻意不主打：

- 多代理并行
- agent team
- 自动组织
- swarm

不是因为这些方向不值得讲，而是因为现在讲会让重点跑偏。

我们当前最需要建立的判断是：

- 什么时候上下文该隔离
- 什么时候子任务该下放
- 什么时候拆分值得

只要这层判断建立起来，后面即使未来扩展到更复杂的多代理协作，读者也不会误解 subagents 的本质。

## 本章最重要的设计取舍

本章要强调两条原则：

- 只有在主上下文明显被噪音拖累时，才值得拆 subagent
- 子代理职责要清晰，不要每个子代理都能做所有事

这两条原则背后其实是同一个思想：

**subagent 是为了降低主线复杂度，而不是增加系统戏剧性。**

## 本章真正要让读者学会什么

读完这一章，读者应该真正理解：

- subagent 的第一价值是上下文隔离，而不是并行炫技
- 研究、实现、审核这类角色边界越清晰，收益越稳定
- 不是所有任务都值得拆 subagent
- 好的子代理输出应该是提炼后的结果，而不是原始噪音

## 本章结论

- subagent 的第一价值是上下文隔离，而不是并行炫技。
- 角色清晰、边界明确的子代理更容易带来稳定收益。
- 如果没有明显的上下文压力，完全可以不拆子代理。
- planning、filesystem context 和 subagents 共同构成了 `Deep Agents` 主线中段的稳定性基础。
