# 15. 可控性：日志、轨迹、审批、人类接管

## 这一章解决什么问题

一个能跑的 agent，不一定是一个能放心交给团队使用的 agent。

这是很多教程和真实项目之间最容易出现断层的地方。

在教程里，我们很容易满足于：

- 看，它会读文件
- 看，它会改代码
- 看，它会跑验证

但一旦进入真实协作环境，问题马上会变成：

- 我怎么知道它到底做了什么
- 它做错的时候我怎么阻止
- 高风险动作能不能先让我确认
- 出问题时我能不能接管

这一章要解决的，就是这些问题。

## 为什么可控性很重要

coding agent 的风险并不只来自“写错代码”，还来自：

- 改错文件
- 执行不合适的命令
- 陷入长循环
- 把大量无关输出塞进上下文
- 在高风险任务上过于自信地继续推进

所以除了任务能力，我们还需要控制能力。

换句话说：

- 第 14 章解决“它做得对不对”
- 第 15 章解决“它是否在可接受的边界内工作”

## 先记住一句话

**一个更聪明但不可控的 agent，通常不如一个足够强且可控的 agent。**

这句话在工程落地里非常现实。

因为团队真正需要的，往往不是：

- 一个偶尔惊艳、偶尔失控的系统

而是：

- 一个持续稳定、能被观察、能被干预的系统

## 核心概念

### 1. 轨迹可见性

用户和开发者需要看见：

- agent 做了哪些步骤
- 调用了哪些工具
- 每一步的结果是什么

否则很难调试，也很难建立信任。

一个完全黑盒的 agent，即使输出偶尔很好，也很难真正进入工程流程。

### 2. 危险动作门控

有些操作天然应该更谨慎，比如：

- 删除文件
- 执行破坏性命令
- 大范围批量修改
- 发布、提交、推送之类的动作

这类动作应该有更高的门槛或人工确认。

### 3. 人类接管能力

再好的 agent 也会遇到超出边界的问题。

系统应该允许：

- 中断
- 修正方向
- 补充约束
- 重新继续

这不是对 agent 的“不信任”，而是对真实工程场景的尊重。

## 为什么日志不是“附加功能”

在很多 demo 里，日志会被当作可有可无的东西。

但在真实项目里，日志和轨迹本身就是系统的一部分。

因为它们直接决定：

- 你能不能调试 agent
- 你能不能复盘失败原因
- 你能不能比较不同策略的效果

所以这一章要建立一个很重要的认识：

**可观测性不是后期再补的装饰，而是 agent 系统的一部分。**

## 为什么高风险动作不能默认自动执行

从工程视角看，有些任务天然更适合自动执行，比如：

- 读文件
- 搜索文本
- 跑轻量验证

但有些动作显然不应该默认自动推进，比如：

- 大范围重构
- 删除目录
- 批量替换
- 发布或部署

这些场景里，真正合理的做法通常不是：

- “永远自动”

而是：

- “先提出计划，再等待确认”

也就是说，可控性不是阻碍 agent，而是帮助它进入真实协作秩序。

## 推荐示例任务

这一章可以设计一个带审批点的示例，比如：

> “在执行大范围重构前，先给出修改计划并等待人工确认。”

这能帮助读者理解：

- 为什么很多高风险动作不该默认自动执行
- 为什么 planning 和 approval 经常会结合出现

## 人类接管为什么不是失败

很多人会下意识觉得：

> 既然是 agent，为什么还要人类接管？

这个问题的潜台词其实是把 agent 理解成：

- 要么全自动成功
- 要么就算失败

但真实项目里，更合理的理解应该是：

- agent 是协作者
- 人类接管是协作机制的一部分

也就是说：

- 能接管
- 能修正
- 能继续

恰恰说明系统更成熟，而不是更差。

## 可观测性建议

这一章可以引入这些实践方向：

- step logging
- structured run records
- tracing
- LangSmith 之类的可观测工具

重点不在于平台本身，而在于：

- agent 不能是黑盒

这一点对教程也很重要，因为教程如果只展示“最终回答”，读者很难真正理解 agent 的运行过程。

## 这一章和第 14 章的关系

第 14 章建立了“验证闭环”，说明任务结束前必须通过环境验证。

第 15 章则进一步建立：

- 任务推进过程本身也必须被观察和控制

也就是说：

- 验证让结果更可信
- 可控性让过程更可信

两者组合起来，agent 才有可能真正走向团队使用。

## 这一章和第 16 章的关系

再往后，第 16 章会讲：

- 什么时候要升级到 `LangGraph`

你会发现，这其实和本章有很强关系。

因为很多“想上 LangGraph”的冲动，背后真正的需求往往不是：

- agent 本身不够强

而是：

- 需要更强的控制
- 需要显式审批节点
- 需要更明确的状态机

所以这一章其实也在为第 16 章铺路。

## 本章最重要的设计取舍

本章最重要的判断是：

- 可控性决定 agent 能否从个人玩具走向团队工具

这意味着我们在设计时要优先思考：

- 哪些步骤该可见
- 哪些动作该门控
- 哪些场景该允许人工中断

而不是只想着：

- 怎么让它再多做一点

## 本章真正要让读者学会什么

读完这一章，读者应该真正理解：

- 轨迹可见、危险动作可控、人类可接管，是工程落地中的关键要求
- agent 不是越自动越好，而是越可协作越好
- 日志和 tracing 是系统能力，不只是调试辅助
- 人类接管不是失败，而是成熟协作的一部分

## 本章结论

- 可控性决定 agent 能否从个人玩具走向团队工具。
- 轨迹可见、危险动作可控、人类可接管，是工程落地中的关键要求。
- 一个“更聪明但不可控”的 agent，通常不如一个“足够强且可控”的 agent。
