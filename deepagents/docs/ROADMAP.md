- `01-17` 章节都已完成一轮系统扩写
- 主线章节已经基本成形
- 当前工作的重点，已经从“补骨架”切换为“统一术语、章节承接和文档站体验”

## 当前阶段重点

当前最值得继续推进的，不再是大面积扩写章节，而是下面这些收口工作：

1. 统一术语
2. 统一章节承接
3. 统一示例映射
4. 优化文档站导航与入口
5. 对工程化章节做第二轮增强

### 第一阶段：整套文档一致性收口

1. 检查 `LangChain v1 / Deep Agents / LangGraph` 表述是否统一
2. 检查 `coding agent / filesystem context / subagents` 等核心术语是否稳定
3. 检查每章结尾的“下一章预告”和章节间承接是否自然
4. 检查章节内的示例路径、源码路径、运行方式是否一致

目标：

- 让教程更像一套连续课程，而不是 17 篇单独文章
- 减少重复表述和局部风格漂移

### 第二阶段：补示例与文档映射

1. 为更多章节补充对应示例入口
2. 在章节中统一标注 `examples/` 和共享 `src/` 的关系
3. 在首页和侧边栏明确推荐阅读顺序

目标：

- 让“文档”和“代码”真正形成闭环
- 降低读者在章节间来回跳转的理解成本

### 第三阶段：第二轮工程化增强

1. `07-project-structure.md`
2. `13-project-knowledge.md`
3. `14-verification-loop.md`
4. `15-observability-and-control.md`
5. `16-when-to-use-langgraph.md`
6. `17-build-your-own-specialized-agent.md`

目标：

- 从“已经能读”进一步走向“更贴近真实项目落地”
- 提升后半段工程化章节的案例密度和判断标准

## 每类章节的完成标准

### 主线支柱章

- 必须和示例或共享代码有清晰对应关系
- 必须写清楚“为什么这样设计，而不是别的方式”
- 必须帮助读者从上一章自然过渡到下一章

### 前置认知章

- 观点要稳定
- 术语要统一
- 问题空间和边界要讲清楚

### 后置扩展章

- 先保证结构完整
- 再逐步补成成稿
- 不抢占主线资源

## 当前阶段结论

当前最佳策略已经不是平均扩写所有文档，而是：

**在主线已经成形的前提下，优先做整套文档的一致性收口、示例映射补齐和工程化章节的第二轮增强。**
