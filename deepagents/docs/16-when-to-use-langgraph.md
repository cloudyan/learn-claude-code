# 16. 什么时候升级到 LangGraph

## 这一章解决什么问题

学到这里，很多读者会自然产生一个问题：

- 既然 `LangGraph` 更底层、更强大，那我是不是应该一开始就用它？

这一章的目标，就是把这个判断讲清楚。

因为如果这个判断不清楚，整套教程到这里很容易出现一种误解：

- 前面都只是过渡
- 最后真正高级的答案一定是 `LangGraph`

但这其实并不准确。

## 为什么不要过早进入 LangGraph

`LangGraph` 很强，但它解决的问题不是“怎么理解 agent”，而是“怎么控制复杂系统”。

如果前面的问题都还没搞清楚，太早进入 `LangGraph` 往往会带来：

- 更多样板代码
- 更强的系统复杂度
- 更高的理解门槛

也就是说，过早进入 `LangGraph` 最常见的后果不是“更专业”，而是：

- 主线注意力被 orchestration 吃掉

这也是为什么这套教程直到第 16 章才专门讲它。

## 先记住一句话

**`LangGraph` 不是更高级的默认答案，而是更强控制需求下的合理答案。**

这句话是整章最重要的判断。

如果你还处在：

- 学 agent 基础
- 做最小 coding agent
- 做单次运行的仓库任务

这些阶段，`Deep Agents` 往往已经足够。

只有当问题真正升级到：

- 更复杂的控制
- 更明确的流程
- 更长期的执行

这时 `LangGraph` 才真正值得上场。

## `Deep Agents` 和 `LangGraph` 解决的问题并不相同

很多人会把它们理解成：

- 一个简单版
- 一个高级版

其实更准确的理解是：

- `Deep Agents` 更像 coding agent 主线 harness
- `LangGraph` 更像 orchestration runtime

它们的关注点不一样。

`Deep Agents` 更关注：

- agent 如何在复杂任务里工作
- 文件系统、subagents、memory、tooling 如何组织

`LangGraph` 更关注：

- 节点怎么编排
- 状态怎么显式流转
- 中断与恢复怎么做
- 审批节点怎么建

所以真正的判断标准不是“哪个更高级”，而是：

- 你当前的问题到底是 agent 问题，还是 orchestration 问题

## 什么时候值得升级

下面这些场景通常是升级信号。

### 1. 需要 durable execution

任务很长，可能中断，之后还需要恢复执行。

这时问题已经不只是“模型能不能继续推理”，而是：

- 系统是否能稳定保存状态
- 是否能在中断后明确恢复到正确阶段

### 2. 需要明确状态机

任务不是简单循环，而是存在清晰阶段和转换条件。

例如：

- 收集上下文
- 生成计划
- 等待审批
- 执行修改
- 跑验证
- 生成 PR 总结

当这些阶段已经明确到需要被系统显式建模时，`LangGraph` 的价值就很高了。

### 3. 需要人工审批节点

例如：

- 先分析
- 再出计划
- 等待人工批准
- 再继续执行

这类流程如果只是硬塞在 prompt 里，最终会越来越脆弱。

而 `LangGraph` 擅长处理这种：

- agent 自主 + 显式人工干预节点

### 4. 需要复杂多阶段 workflow

比如：

- 先收集上下文
- 再规划
- 再修改
- 再验证
- 最后生成 PR 总结

如果你已经开始明确地把系统拆成多个稳定阶段，那就说明 orchestration 问题正在上升。

### 5. 需要长期运行和恢复

这类系统往往已经接近“agent platform”而不只是“agent script”。

一旦进入这个阶段，很多原本靠 prompt 和习惯维持的逻辑，都需要变成显式系统结构。

## 一个更实用的判断方式：先拿典型场景逐个问

如果你在真实项目里总是犹豫“我是不是该上 `LangGraph` 了”，一个更实用的做法不是反复抽象思考，而是：

- 先挑几个典型任务
- 逐个判断它们到底缺的是 agent 能力，还是 orchestration 能力

例如：

- 单次仓库修复任务
- 带审批点的多阶段任务
- 可恢复的长时任务

当你把问题放回具体场景，边界通常会清楚很多。

这也是本章示例最适合采用的方式：

- 不是抽象讲框架
- 而是对多个场景逐个做判断

## 什么时候不必升级

如果你当前主要做的是：

- 小到中等复杂度的 coding task
- 单次运行的仓库任务
- 以 agent 自主推进为主的工作流

那么 `Deep Agents` 往往已经够用。

这点很重要，因为很多系统其实还没有复杂到值得引入更重的 orchestration。

如果太早下沉，常见代价是：

- 理解成本上升
- 维护成本上升
- 教程与项目复杂度同时上升

## 为什么很多“想升级”的冲动其实是可控性需求

在真实工程里，很多人会说：

> 我是不是该上 LangGraph 了？

但再追问一步，背后真正的诉求往往是：

- 我想加审批
- 我想加可恢复执行
- 我想让状态显式一点
- 我想把主流程拆节点

你会发现，这些都不是“模型不够聪明”的问题，而是：

- orchestration 不够强

所以本章真正想帮助读者建立的判断是：

不要因为“想更高级”而升级，而要因为“控制问题真的出现了”而升级。

## 为什么这章必须放在后面

如果这章放在教程前面，读者很容易把注意力放在：

- graph 怎么建
- node 怎么拆
- state 怎么定义

这样就会过早偏离：

- agent 怎么工作
- coding task 怎么推进
- 为什么 `Deep Agents` 适合主线

所以这章必须放在后面。

只有当前面这些都已经理解之后，读者才能真正判断：

- 我到底是不是已经走到了需要 `LangGraph` 的阶段

## 本章最重要的设计取舍

这一章最重要的观点是：

- `LangGraph` 不是更高级的默认答案
- 它是更强控制需求下的合理答案

换句话说：

- 当 agent autonomy 已经足够，但 orchestration 不够时，才是该进入 `LangGraph` 的时候

## 本章真正要让读者学会什么

读完这一章，读者应该真正理解：

- 是否使用 `LangGraph`，取决于 orchestration 复杂度，而不是学习冲动
- `Deep Agents` 和 `LangGraph` 解决的问题层级不同
- 审批节点、恢复执行、显式状态机，是最典型的升级信号
- 大多数主线阶段，`Deep Agents` 已经足够承载 coding agent

## 本章结论

- 是否使用 `LangGraph`，取决于 orchestration 复杂度，而不是学习冲动。
- 大多数教程主线阶段，`Deep Agents` 已经足够承载 coding agent。
- 当系统需要 durable execution、审批节点、复杂状态机时，再下沉到 `LangGraph` 更合理。
